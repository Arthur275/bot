# ETH 运行观察面板产品设计规格

## 1. 产品定位

这个 dashboard 不是交易所页面、不是营销页面、不是人工控制台。

它的定位是：

```text
ETH 自动化实盘链路的只读运行驾驶舱
```

用户打开页面后，第一眼要知道：

```text
系统现在活没活
链路卡在哪一层
为什么现在没有下单
如果下单链路有动作，bot 和 worker 做了什么
因子和量化数据是否新鲜
审查报告是否发现额外解释性风险
```

页面要服务的是“持续观察”和“快速定位因果”，不是展示所有字段。

## 2. 设计原则

### 2.1 清晰优先

所有信息必须围绕链路组织：

```text
样本采集与因子治理
  -> 量化市场判断
  -> Bot 下单链路
  -> 决策审查报告
```

不能把字段平铺成数据字典。

### 2.2 只读边界

页面不提供任何实盘控制能力。

禁止：

```text
真实下单开关
人工放行按钮
强制刷新因子按钮
强制生成 candidate package
绕过 veto / kill switch
修改 sizing
修改 factor lifecycle
```

允许：

```text
手动刷新页面数据
复制 run_id / package_id
展开或折叠明细
过滤本地展示的日志
```

### 2.3 中文优先

页面展示必须默认中文。

规则：

```text
字段名中文
状态中文
空值显示“暂无”
错误显示“不可用”或“缺失”
英文 key 只在技术明细里保留
reason code 要优先显示中文解释，必要时附带原始 code
```

### 2.4 高信息密度但不压迫

这个页面面向运行观察，不适合大 hero、大插图、大面积留白。

需要：

```text
主指标醒目
次指标紧凑
明细可扫读
日志可换行
状态色统一
```

## 3. 信息架构

页面分为五层：

```text
1. 全局顶部状态
2. 链路健康条
3. 四个主面板
4. 明细抽屉 / 折叠区
5. 异常和日志区域
```

四个主面板：

```text
1. 样本采集与因子治理
2. 量化市场判断
3. Bot 下单链路
4. 决策审查报告
```

## 4. 第一屏目标

桌面端第一屏必须能看到：

```text
全局状态
四层链路健康
量化当前 action / direction / risk
Bot 当前 candidate / worker 状态
kill switch 状态
数据新鲜度
```

第一屏不能被长文本撑爆。

移动端第一屏必须能看到：

```text
全局状态
链路健康条
当前最关键阻断原因
```

## 5. 页面总布局

### 5.1 桌面端布局

建议 1440px 以上：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部：ETH 运行观察面板 / 更新时间 / 全局状态 / 手动刷新       │
├──────────────────────────────────────────────────────────────┤
│ 链路健康条：采集 -> 量化 -> Bot -> Worker -> 审查             │
├──────────────────────────────┬───────────────────────────────┤
│ 样本采集与因子治理           │ 量化市场判断                  │
├──────────────────────────────┼───────────────────────────────┤
│ Bot 下单链路                 │ 决策审查报告                  │
└──────────────────────────────┴───────────────────────────────┘
```

推荐 2x2，不再使用当前三列。

原因：

```text
三列会让中间量化区过长，两侧信息不均衡。
2x2 更符合四块业务链路，也方便加入审查报告。
```

### 5.2 中等屏幕布局

980px 到 1280px：

```text
顶部
链路健康条
量化市场判断
样本采集与因子治理
Bot 下单链路
决策审查报告
```

量化面板优先级最高，应提前。

### 5.3 移动端布局

720px 以下：

```text
顶部状态
关键阻断原因
链路健康条
量化市场判断
Bot 下单链路
样本采集与因子治理
决策审查报告
```

移动端按“用户最想知道为什么没下单”的顺序排。

## 6. 全局顶部状态

顶部区域包含：

```text
标题：ETH 运行观察面板
副标题：只读观察自动化实盘链路
更新时间
刷新状态
真实下单模式：禁用 / dry-run / submit-enabled
kill switch：关闭 / 开启
```

顶部不使用大标题 hero。

视觉规则：

```text
高度 64-80px
左侧标题
右侧状态胶囊 + 刷新按钮
移动端上下排列
```

## 7. 链路健康条

链路节点：

```text
样本采集
量化判断
Bot 调度
执行 worker
决策审查
熔断开关
```

每个节点显示：

```text
名称
状态
更新时间 / age
简短问题
```

节点状态：

| 状态 | 中文 | 颜色 |
| --- | --- | --- |
| running | 运行中 | 绿 |
| fresh | 新鲜 | 绿 |
| watch | 观察 | 蓝 |
| degraded | 降级 | 黄 |
| stale | 过期 | 黄 |
| blocked | 阻断 | 红 |
| veto | 否决 | 红 |
| missing | 缺失 | 灰 |
| unavailable | 不可用 | 灰 |

链路条要能表达“卡在哪”：

```text
采集正常 -> 量化 veto -> Bot 等待 -> Worker skipped
```

## 8. 面板通用结构

每个主面板统一结构：

```text
面板标题
状态 badge
一句话摘要
3-5 个主指标
次级状态标签
明细区
日志 / 长文本区
```

禁止：

```text
卡片里再套大卡片
无层级的 dl 列表堆满整块
所有字段同等字号
长文本直接进入主指标区
```

## 9. 样本采集与因子治理面板

### 9.1 主问题

这个面板回答：

```text
因子数据有没有持续采集？
lookup 是否可用？
治理是否认为因子可用？
有没有样本不足、过期、净期望为负？
```

### 9.2 主指标

主指标最多 5 个：

```text
样本数
观察数
lookup 行数
因子值数量
治理状态
```

### 9.3 次指标

```text
lookup version
lookup freshness
DuckDB 状态
bot scheduler samples
factor governance generated_at
```

### 9.4 明细展示

因子治理行用 compact table，而不是一条条长列表。

列：

```text
因子
分桶
等级
生命周期
效果
样本
胜率
净期望
状态
```

表格最多展示 6-10 行，更多后续可加“展开”。

### 9.5 风险代码

风险代码和降级标记要分组：

```text
样本问题
新鲜度问题
research 问题
lookup 问题
执行阻断问题
```

长 code 显示方式：

```text
中文解释优先
原始 code 小字灰色显示
自动换行
不允许横向撑出
```

## 10. 量化市场判断面板

### 10.1 主问题

这个面板回答：

```text
量化现在想做什么？
方向是什么？
为什么没有进入 bot 下单？
风控是 pass / degraded / veto 还是 blocked？
```

### 10.2 主指标

```text
动作
方向
风控状态
仓位档位
置信度
```

动作中文映射：

| 原始值 | 中文 |
| --- | --- |
| entry_long | 开多 |
| entry_short | 开空 |
| small_probe | 小探针 |
| wait | 等待 |
| reduce | 减仓 |
| exit | 平仓 |
| observe_only | 仅观察 |
| paper_only | 纸面 |

### 10.3 核心摘要

reasoning summary 显示为“量化摘要”。

规则：

```text
最多显示 3-5 行
超过高度可滚动或折叠
必须换行
不能穿到右边面板
```

### 10.4 因子证据

分三组：

```text
支持开仓
反对开仓
否决开仓
```

每个 chip：

```text
中文名
原始 code
来源
```

否决因素必须视觉上比支持因素更醒目。

### 10.5 Research 健康

Research 区域不要抢主指标。

展示：

```text
状态
新鲜度
是否可用于决策
数据时间
刷新周期
原因代码
```

## 11. Bot 下单链路面板

### 11.1 主问题

这个面板回答：

```text
Bot 收到 handoff 了吗？
有没有 candidate package？
真实下单 gate 是否允许？
worker 是 submitted / skipped / blocked？
当前持仓是什么？
```

### 11.2 主指标

```text
执行状态
持仓状态
候选包状态
worker 最新状态
真实提交模式
```

### 11.3 Candidate package

展示结构：

```text
package_id
action
direction
generated_at
expires_at
gate_allowed
commands
```

如果缺失，要显示缺失原因：

```text
handoff 缺失
execution_allowed=false
risk_filter veto
research_not_ready
candidate expired
```

### 11.4 Worker events

worker events 用日志列表：

```text
时间
状态
动作
原因
audit id
```

最多展示最近 8-20 条。

日志区域必须：

```text
固定最大高度
内部滚动
长 reason 自动换行
```

## 12. 决策审查报告面板

### 12.1 定位

审查报告是解释层，不是放行层。

必须显示固定提示：

```text
审查报告仅供解释和复盘，不参与自动下单。
审查清晰不代表允许下单，最终以下单链路和风控结果为准。
```

### 12.2 主指标

```text
审查状态
审查模式
来源 run_id
handoff 新鲜度
耗时
```

状态：

```text
clear -> 清晰
watch -> 观察
needs_attention -> 需关注
unavailable -> 不可用
```

不要使用“通过”。

### 12.3 数据源质量

展示 `data_source_quality`：

```text
handoff
factor_lookup
factor_summary
risk_report
candidate_package
worker_audit
outcome_samples
```

缺关键数据源时必须显示黄色或灰色，不得显示绿色。

### 12.4 审查内容

分区：

```text
多头理由
空头理由
风险发现
执行链路发现
治理复查建议
未解决问题
```

每区最多显示 3-5 条，后续再加展开。

### 12.5 视觉边界

审查面板不要和量化面板使用同样的“放行”视觉。

建议：

```text
审查 clear 使用蓝绿色或蓝色，不使用纯绿色大成功态
needs_attention 使用黄色或红黄
unavailable 使用灰色
```

## 13. 视觉系统

### 13.1 推荐浅色主题

优先做浅色专业面板。

理由：

```text
中文可读性更稳
长时间观察眼压低
状态色更清楚
不会像黑色交易所页面那样压迫
```

推荐色：

```text
页面背景：#f5f7fb
面板背景：#ffffff
次级背景：#f8fafc
边框：#d8e0eb
强边框：#b7c4d6
主文字：#172033
次文字：#627083
弱文字：#8a97a8
绿色：#16875a
蓝色：#2563eb
黄色：#b7791f
红色：#c2410c
灰色：#64748b
```

### 13.2 不建议的风格

避免：

```text
紫蓝渐变主导
大面积深蓝 / 深黑
装饰性光斑
营销 hero
圆角过大的卡片
过多阴影
所有 badge 都高饱和
```

## 14. 字体系统

字体栈：

```css
font-family:
  "Microsoft YaHei UI",
  "Microsoft YaHei",
  "PingFang SC",
  "Noto Sans CJK SC",
  "Segoe UI",
  Arial,
  sans-serif;
```

字号建议：

```text
页面标题：20-22px
面板标题：16-17px
主指标数值：20-24px
主指标标签：12px
正文：13px
明细：12px
日志：12px
辅助说明：11px
```

规则：

```text
不要用 viewport width 缩放字体
letter-spacing = 0
按钮文字不能挤压
中文行高 1.45-1.65
```

## 15. 组件规范

### 15.1 Badge

Badge 用于状态，不用于普通字段。

形态：

```text
圆角 999px
高度 24-28px
文字 11-12px
边框 + 浅底色
```

状态色规则：

```text
绿色：运行正常、数据新鲜、worker submitted
蓝色：观察、审查 clear、信息状态
黄色：降级、过期、watch、缺次要数据
红色：veto、blocked、error、kill switch on
灰色：missing、unavailable、disabled
```

### 15.2 主指标卡

每个面板 3-5 个。

规则：

```text
固定高度 72-84px
文本居中
数值大
标签小
不要因为内容变长导致高度变化
```

### 15.3 明细表

明细使用两种形态：

```text
key-value grid：少量字段
compact table：因子治理、数据源质量、worker events
```

不要所有东西都用 `dl`。

### 15.4 Chip

Chip 用于 reason code / factor code。

规则：

```text
最多宽度 100%
自动换行
支持中文解释 + 原始 code 两层文字
不要全大写
```

### 15.5 日志盒子

用于：

```text
worker events
reasoning summary
长 transition reason
审查发现
```

规则：

```text
最大高度
内部滚动
等宽或半等宽可选
不撑破布局
```

## 16. 文案规范

### 16.1 空值

```text
null / undefined / "" -> 暂无
[] -> 暂无
missing artifact -> 缺失
service unavailable -> 不可用
stale -> 已过期
```

### 16.2 模式文案

```text
dry_run -> 模拟执行
submit_enabled -> 真实提交已启用
disabled_by_kill_switch -> 熔断禁用
```

### 16.3 审查文案

禁止：

```text
审查通过
允许下单
AI 已确认安全
```

允许：

```text
审查清晰
未发现额外解释性风险
需要关注
审查不可用
```

## 17. 数据状态规则

### 17.1 不闪屏

5 秒刷新时：

```text
保留旧数据
顶部显示刷新中
失败时显示 error banner
不清空页面
```

### 17.2 数据新鲜度

所有主要数据源要显示 age：

```text
factor ingest age
quant handoff age
bot heartbeat age
worker audit age
review report age
```

### 17.3 缺数据策略

缺数据时：

```text
展示“暂无/缺失/不可用”
展示影响范围
不要隐藏整个面板
```

## 18. 响应式规则

### 18.1 桌面端

```text
最大内容宽度 1600-1720px
2 列主网格
面板最小宽度 0，避免撑爆
```

### 18.2 平板

```text
单列或量化优先
链路健康条横向滚动或两列换行
```

### 18.3 手机

```text
单列
顶部居中
主指标 2 列或 1 列
表格改为卡片行
日志内部滚动
```

## 19. 性能要求

前端继续原生 HTML/CSS/JS。

理由：

```text
页面是本地只读观察面板
交互不复杂
无需构建链
启动稳定性高
体积小
```

性能规则：

```text
5 秒轮询 /api/overview
server 端继续 1 秒 cache
前端不要重复创建大量不可见 DOM
worker events 限制最近 20 条
reason/chips 限制展示数量
长历史不要一次性渲染
```

## 20. 可访问性和可读性

要求：

```text
颜色不能作为唯一状态表达
badge 必须有文字
对比度足够
按钮有 hover/focus
表格在移动端不横向撑爆
```

## 21. 当前页面主要问题

当前版本问题：

```text
1. 三列布局和业务四块不匹配。
2. 深色主题压迫，中文长时间阅读不舒服。
3. 面板内部层级弱，字段像堆列表。
4. 量化 reasoning 和长 code 仍然容易显得杂乱。
5. 审查报告没有进入视觉结构。
6. 状态色语义不够统一。
7. 主指标与明细之间视觉差距不够。
8. 缺少“当前卡在哪”的全局因果摘要。
```

## 22. 改版实施顺序

### Phase 1：信息架构重排

```text
新增四块布局结构
顶部全局状态
链路健康条
审查报告面板占位
不改 API 语义
```

### Phase 2：视觉系统重做

```text
切浅色主题
统一 badge / metric / table / chip / log box
中文字体和字号重设
```

### Phase 3：字段中文化和分层

```text
主指标只放关键字段
次级字段进明细区
长 reason 进日志盒子
reason code 中文解释优先
```

### Phase 4：响应式和溢出治理

```text
桌面 2x2
平板单列优先量化
手机单列
长文本和表格不撑爆
```

### Phase 5：真实页面验证

```text
启动 dashboard
检查 /api/overview
桌面截图
移动端截图
长文本样本压测
空数据样本压测
```

## 23. 测试验收

必须补或更新测试：

```text
DOM contract：所有 JS 引用 id 存在
HTTP API：/、/app.js、/styles.css、/api/overview 可访问
中文契约：核心面板标题和关键状态中文存在
安全契约：app.js 不使用 innerHTML 渲染 runtime 数据
兼容契约：不使用 replaceChildren
响应式契约：CSS 包含 1280 / 980 / 720 断点
溢出契约：长文本容器有 overflow-wrap / word-break / min-width: 0
审查边界契约：页面包含“审查不参与自动下单”文案
```

手工验收：

```text
桌面 1440px：四块 2x2，第一屏能看清链路状态
桌面 1920px：内容不散，不超宽
平板 1024px：量化面板优先，不横向滚动
手机 390px：无文字穿出，无按钮挤压
长 transition reason：不穿到相邻面板
空数据：不显示 undefined/null
刷新失败：保留旧数据并显示错误条
```

## 24. 最终验收标准

改版完成后，页面应该达到：

```text
一眼知道系统是否运行
一眼知道当前是否可能下单
一眼知道阻断发生在哪层
30 秒内能定位为什么没下单
长文本不会撑爆布局
中文字段完整
审查报告不会被误解为交易放行
```

如果页面只是“更好看”，但不能更快回答“为什么没下单”，改版就不算成功。
